# 로그 분석 보고서 (2026-01-05)

## 1. 시스템 상태 요약
* **전체 상태**: 시스템이 정상적으로 부팅되었으며, 모든 주요 노드(Driver, EKF, SLAM, Lidar)가 성공적으로 실행되었습니다.
* **종료 상태**: 사용자가 `Ctrl+C`로 종료할 때까지 약 3분 30초간 실행되었으며, 모든 프로세스가 정상적으로 종료되었습니다.

## 2. 주요 로그 분석

### A. 정상 작동 확인
* **드라이버**: `Rosmaster hardware initialized successfully` - 하드웨어 연결 성공.
* **라이다**: `SLLidar health status : OK` - 라이다 센서 정상 작동.
* **IMU 캘리브레이션**: `Gyros calibration complete!` - 자이로 바이어스 보정 성공.
* **SLAM**: `Trying to create a map of size...` - 지도가 지속적으로 확장/업데이트되고 있음 (정상).

### B. 경고 및 에러 메시지 분석

#### 1. `Message Filter dropping message ... queue is full`
* **현상**: 실행 초반(약 10초간) 라이다(`laser`)와 오도메트리(`odom`) 메시지가 일부 버려짐.
* **원인**: 시스템 부팅 직후 CPU 부하가 높거나, EKF가 TF 트리를 구성하는 데 시간이 걸려 일시적으로 데이터 처리가 밀림.
* **판단**: **무시 가능**. 초반 이후에는 해당 메시지가 사라지고 지도 생성 로그(`Trying to create a map...`)가 주를 이루므로 시스템이 안정화된 것으로 보입니다.

#### 2. `The link ... has unrealistic inertia`
* **현상**: Rviz에서 휠 링크의 관성 모멘트가 비현실적이라는 에러 출력.
* **원인**: URDF 파일의 관성 파라미터가 물리 엔진(Gazebo 등) 기준에 맞지 않음.
* **판단**: **무시 가능**. 현재 실제 로봇을 구동 중이므로 시뮬레이션용 물리 파라미터는 주행에 영향을 주지 않습니다.

#### 3. `minimum laser range setting ... exceeds capabilities`
* **현상**: SLAM Toolbox에서 라이다 최소 거리가 0.0m로 설정되었으나 센서 스펙은 0.1m임.
* **판단**: **무시 가능**. SLAM 알고리즘이 자동으로 센서 스펙에 맞춰 0.1m 이내의 데이터는 무시합니다.

#### 4. `the timestamp on the message is earlier than ... transform cache`
* **현상**: 종료 직전(약 1767589886 시점)에 다시 메시지 드롭 현상 발생.
* **원인**: 시스템 부하 또는 일시적인 통신 지연으로 추정되나, 지속적이지 않음.

## 3. 결론 및 제언
* **시스템 건전성**: **매우 양호**. 초반의 일시적인 메시지 드롭을 제외하면 치명적인 오류 없이 SLAM이 수행되고 있습니다.
* **지도 작성**: `Trying to create a map of size 140 x 155` 로그를 볼 때, 로봇이 이동함에 따라 지도가 정상적으로 확장되고 있습니다.
* **향후 조치**: 현재 상태에서 로봇을 천천히 주행하며 지도를 완성하시면 됩니다. 만약 지도가 찌그러진다면 이는 소프트웨어 에러가 아니라 **IMU 회전 보정(튜닝)**의 정밀도 문제일 확률이 높습니다.
